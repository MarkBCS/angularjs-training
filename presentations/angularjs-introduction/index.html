Part 1

Background and a High-level Overview

Background

AngularJS1

  • Open-source JavaScript framework, maintained by Google
  • JavaScript MVC Framework for building Single-Page Applications (SPAs)
  • Notable emphasis on testability and embeddability
  • A framework: you either use it in full or not at all
  • MIT licensed

Similar Libraries and Frameworks

  • Backbone.js
    Minimalistic, low-level, make-your-own-stack library
  • Ember.js
    Minimal boilerplate, opinionated, "developer-friendly", very Angularish
  • Knockout
    MVVM framework. Declarative bindings, automatic UI refresh
  • Polymer
    Component-based application library (pre-alpha)

Google trending skyrocketing since early 2013
(230% increase in 01-07/2013)

Design Overview

The Zen of Angular2

"Angular is built around the belief that declarative code is better than imperative when it comes to building UIs and wiring software components together, while imperative code is excellent for expressing business logic."

Design goals2

  • Decouple DOM manipulation from app logic (→ testability)
  • Regard application testing as equal in importance to application writing (testing difficult affects code structuring)
  • Decouple server side and client side (→ reusability, testability)
  • Guide developers (UI design → business logic → testing)

Use Cases

Angular Sweet Spot

"Angular was built for the CRUD application in mind. Luckily CRUD applications represent the majority of web applications." 2

Application examples:

  • The canonical AngularJS CRUD apps
  • http://builtwith.angularjs.org/project/Project-Time-Recorder


Use of AngularJS for GUI editors and games is specifically discouraged 2

Real-time Web Applications

Extend Angular's two-way data binding to server-side. Still a little bit experimental, but doable:

  • Hook up to socket.io interface with Angular Socket.io Seed
  • Use a Firebase backend with AngularFire


Application examples: chatroom, real-time TODO app


...But maybe just for real-time CRUD applications

Counter-Arguments

Claims (e.g. [6])

Angular is:

  • large and monolithic,
  • has a huge API, and
  • requires you to wrap code to proprietary "directives" and "filters"

→ Know the implications,
but don't worry too much!

Potential deal-breakers:

  • You rely on a lot of non-Angular libs that are hard to integrate
  • Strict size, performance or interoperability requirements
  • Requirements on component reusability outside Angular
  • Application domain outside CRUD

Bootstrap and Initialization

Bootstrapping

<html ng-app>

Minimal Angular app = put ng-app somewhere + load angular.js
(If IE7 support is required add id="ng-app" too)

Automatic Initialization

Automatic Initialization

ngApp in Detail

  1. Automatically initialize Angular after document has loaded
  2. Look for ng-app attribute in the DOM hierarchy
  3. If a module was referenced, load it (here: myModule)
  4. Create application injector
  5. Compile DOM that resides inside the element containg ng-app attribute

Example

jsfiddle.net/jsalonen/Z9XC9/

<script type="text/javascript">
    var myApp = angular.module('myApp', []);
</script>
<div ng-app="myApp">
  1, 2, ..., 10 = {{1+2+3+4+5+6+7+8+910}}
</div>
<div class="some-container">
  1 + 2 = {{1+2}}
</div>

Manual Initialization

Omit ng-app to manually orchestrate initialization

<!doctype html>
<html xmlns:ng="http://angularjs.org">
  <body>
    Hello {{'World'}}!
    <script src="http://code.angularjs.org/angular.js"></script>
    <script>
      angular.element(document).ready(function() {
        angular.module('myApp', []);
        angular.bootstrap(document, ['myApp']);
      });
    </script>
  </body>
</html>
  • element = angular.element ("Wraps a raw DOM element or HTML string as a jQuery element")
  • angular.bootstrap invokes compile to turn the element into an executable, bi-directionally bound application.

$ Prefix Indicates Angular Namespace

"To prevent accidental name collision, Angular prefixes names of objects which could potentially collide with $. Please do not use the $ prefix in your code as it may accidentally collide with Angular code."
  • A handy convention to keep you on track what comes from Angular
  • Don't break the convention. Don't prefix your own objects with $.
  • Note that $ object usually refers to jQuery, not Angular

Data Binding Glues Models to Views

  • Initialize model variables with ng-init directive
  • Bind variables to view with ng-model directive
  • No distinct model object - data is stored inside scope along with functions

Controllers

  • Controller: a JavaScript construction function that is used to augment the scope
  • The augmented scope inherits all properties from upper-level scope (top-level is $rootScope)
  • ng-controller: directive that calls the construction function for the named controller
  • Controller-augmented scope is only visible inside the HTML block that declares ng-controller

    Using Controllers Correctly

    Do not use controllers for:

    • Direct DOM manipulation (→ Angular's data binding instead)
    • Input formatting (→ angular form controls)
    • Output formatting (→ filters)
    • Sharing stateless or stateful code (→ services)
    • Managing life-cycle of other components (→ injector)

    Modules

    Modules - What and Why?

    • Modules are a declarative approach to the ubiquitous (and imperative) "main" method
    • Angular applications are defined as Modules
    • Also: package third-party code as reusable modules
    • Modules improve testability

    Recommended Setup

    While any number of components (controllers, services, directives, etc.) can be bound to a single module, it is recommended to break your app into multiple modules:

    • A service module, for service declaration
    • A directive module, for directive declaration
    • A filter module, for filter declaration
    • And an application level module which depends on the above modules, and which has initialization code.

    Module Loading & Dependencies

    Two kinds of blocks:

    • Configuration blocks - gets executed during the provider registrations and configuration phase.
    • Run blocks - get executed after the injector is created and are used to kickstart the application.

    Configuration Blocks

    In the previous example we called:

    myApp.controller('NameListCtrl', function($scope) { ... });
    It is, in fact, a convenience method for calling
    controllerProvider in config:
    angular.module('myApp', []).
  config(function($controllerProvider) {
    $controllerProvider.register('NameListCtrl', function($scope) { ... });
  }
);

    jsfiddle.net/jsalonen/aW49w

    Configuration Block vs. Convenience Methods

    More examples:

    angular.module('myModule', []).
  value('a', 123).
  factory('a', function() { return 123; }).
  directive('directiveName', ...).
  filter('filterName', ...);

    Is same as:

    
angular.module('myModule', []).
  config(function($provide, $compileProvider, $filterProvider) {
    $provide.value('a', 123);
    $provide.factory('a', function() { return 123; });
    $compileProvider.directive('directiveName', ...);
    $filterProvider.register('filterName', ...);
  });

    Expressions

    Expression

    • JavaScript-like code snippets, mainly used for data binding
    • Usually placed in bindings like: {{ expression }}
    • Are re-evaluated when bindings change

    Different from JavaScript Expressions

    • Evaluated against scope object (not window)
    • Silently handle undefined and null values
    • You cannot use control flow statements
    • Parsed with $parse, not with eval (=safer)

    Expression Examples

    Some simple math with values from scope:

    {{ savings + income - moneySpent }}
    {{ 3.14 * radius * radius }}

    String concatenation with null / undefined values:

    {{ "hello" + null + undefined + "world!" }}
    → "hellonull world!"

    Additional Notes

    • If you really need window object, you can use $window
    • If you really need eval, you can use $eval
    • Don't overdo expressions:
      • Filters for data manipulation
      • Directives for reusable components
      • Controllers for logic

    Filters

    • Filters perform data transformation (similar in spirit to UNIX pipe (|) filters
    • Filters can and are often chained
    • Examples:
      • filter a list by search string
      • format data according to a data type or locale
      • convert data into a specific format (e.g. json filter)

    Using Filters in View Templates

    Typical usage: apply filters to expressions in view templates:

    {{ expression | filter }}

    Filter chaining:

    {{ expression | filter1 | filter 2 }}

    With arguments

    {{ expression | filter1:arg1:arg2 }}

    Examples of Built-in Filter Usage

    jsfiddle.net/jsalonen/QH9yw/

    Uppercasing:

    {{name|uppercase}}

    Currency formatter:

    {{salary|currency:"€"}}

    More built-in filters: lowercase, date, filter, json, limitTo, number, orderBy

    Practical Example:
    Simple filter-based search

    jsfiddle.net/jsalonen/9c8Tz

    Template (with search filter):

    <div ng-controller="ParticipantsCtrl">
  <p>Search participants: 
    
  <ol>
    <li ng-repeat="participant in participants | filter:search">
      {{participant.name}}
    </li>
  </ol>
</div>

    Controller (with orderBy filter):

    function ParticipantsCtrl($scope, $filter) {
  var participants = [{name: 'Jaakko'}, {name: 'Aaro'}, ...];
  $scope.participants = $filter('orderBy')(participants, 'name');
}

    Directives

    Directive

    • A behaviour or DOM transformation, which is triggered by the presence of a custom attribute, element name, class name or comment
    • Allows you to extend the HTML vocabulary in a declarative fashion.
    • Multiple directives have already been introduced
      (can you recall them?)
    • Simple example: ngHide = hide/show element based on expression
    • A more advanced example: make content editable with contentEditable="true" attribute.

    Directive Matching

    Simple directive example: 

    <input ng-click="action()">

    ng-click is matched with ngClick directive due to normalization:

    • Strip x- and data- from element/attribute name prefix
    • Convert ., - and _ delimited names to camelCase


    Best practice: use dash-delimited format in your names

    Directive Types

    Angular's $compile can match directives in three ways. From:

    • Element names (<my-dir></my-dir>)
    • Attributes (<h1 my-dir="exp"><h1>)
    • Comments (<!-- here is my-dir -->)


    Prefer tag name directives

    Example Attribute Directive:
    centralize

    jsfiddle.net/jsalonen/A3dJW

    myApp.directive('centralize', function() {
 return {
  restrict: 'A',
  link: function($scope, element, attrs) {
   var elemWidth = element.context.offsetWidth;
   var elemHeight = element.context.offsetHeight;
   var top = Math.max(0, (($(window).height() - elemHeight) / 2) + $(window).scrollTop());
   var left = Math.max(0, (($(window).width() - elemWidth) / 2) + $(window).scrollLeft());
   $(element).css({
    position: 'absolute',
    'z-index': 1000,
    margin:0,
    top: (top > 0 ? top : 0)+'px',
    left: (left > 0 ? left : 0)+'px'
   });
  }
 }
});

    When do I need a directive?

    "If you end writing $(element) somewhere in your controller then its a indication that you need a directive."

    Services

    Angular Services

    • Angular services = substituable objects that are wired together using dependency injection

    • Use services to organize and share code across your application
    • General-purpose services are provided by Angular

    Using services

    jsfiddle.net/jsalonen/jtge8/

    Example services: $timeout and $window

    Template:

    <div ng-app="myApp"><div ng-controller="QueueCtrl">
  <input type="submit" value="Click to start queuing" ng-click="enterQueue()">
</div></div>

    Code:

    angular.module('myApp', []).
  controller('QueueCtrl', ['$scope', '$timeout', '$window',
    function($scope, $timeout, $window) {
    $scope.enterQueue = function() {
      $timeout(function() {
        $window.location.href = 'http://www.gofore.com';
      }, 5000);      
    };
  }]);

    Creating Services

    Applications may (and often should) create their own services:

    1. Register service with service factory function
    2. Specify dependencies
    3. Use as required

    Example Service: TaxCalculatorService

    jsfiddle.net/jsalonen/saG3p

    myApp.factory('TaxCalcService', function() {
    return {
      'calculateTax': function (income) {
        if(income<10000) return income*0.1;       
        if(income<20000) return income*0.2;        
        return income*0.3;
      }      
    };
  })

    Services vs. Factories

    Both service and factory methods allow you to create injectable singletons.

    However:

    myApp.service('serviceA', function a() { ... })

    Returns instance of the function (new function a()) 

    myApp.factory('serviceB', function b() { ... })

    Returns the value that is returned when invoking b

    Server-side Communications

    $http

    • Rudimentary service for HTTP calls
    • Supported methods: GET, HEAD, POST, PUT, DELETE
    • Ability to use JSONP
    • In practice very similar to jQuery's ajax method

    • Higher-level service for accessing RESTful server-side resources

    Providers

    • Provider = the way how registration of various objects for injector is handled in AngularJS
    • Value, Factory, Service and Constant methods are just syntactic sugar on top of provider recipe
    • More details: at Angular guide on Providers

    Forms

    • A set of features for developing forms is provided
    • Included out of the box: form controller and validation facitilities
    • Combine with e.g. $resource to implement form submission

    Localization

    • Support for datetime, number and currency localization via filters
    • Pluralization support via ngPluralize directive
    • Localizable Angular components depend on $locale service

    Ensuring Browser Compatibility

    Internet Explorer Compatibility

    • Add id="ng-app" to the element containing ng-app
    • If you use custom element tags (<ng:view>), extra boilerplate need to be added (IE < 8)
    • Polyfill JSON.stringify (IE7 and below)
    • Use ng-style tags for styles containing expressions (IE < 11)


    For details see http://docs.angularjs.org/guide/ie

    Ensuring JavaScript Code Quality

    ECMAScript 5 Strict Mode

    • A new feature in ECMAScript 5 that allows you to place a program, or a function, in a “strict” operating context.
    • Catches common coding "bloopers", throwing exceptions
    • Prevents or throws errors on unsafe actions
    • Disables confusing or poorly designed JavaScript features

    "use strict"

    Enable for whole script by adding in the beginning of the file:

    "use strict"

    Can be also used for individual functions:

    function myFunc() {
  "use strict";
}

    Catch: requires IE 10 or later (but does not break on older versions)

    Linting

    • Linting = checking your source code for style guideline compliance and suspicious usage of language
    • Especially important in dynamically typed languages like JavaScript where you lack the whole compile-time checking safety net
    • In addition works as a style guideline compliance checker

    jslint, jshint

    • Commonly used JavaScript linting tools
    • Choose one, add it to your build/CI-pipeline
    • You may end up disagreeing with the defaults
    • However: study to linting errors in depth before tweaking

    Minifying Angular Code

    Be Careful with Dependency Injection

    Be careful when using dependency injection in this fashion:

    angular.module('myApp').controller('MyControl', function ($scope, $http) {
  ...
});

    Minifiers can mangle object names, which break dependency injection (e.g. $scope → s)

    To stay minification-safe, write:

    angular.module('myApp').controller('MyControl', ['$scope', '$http', function ($scope, $http) {
  ...
})

    Minifiers don't mangle string values, ensuring injections work even with renamed parameter variables

    AngularJS Pre-minifier

    • You may still prefer or need to use the shorter form
    • Work around the problem by using a preminifier (ngmin)
    • Integrate to your build e.g. with grunt-ngmin or gulp-ngmin
