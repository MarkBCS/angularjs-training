Part 1

Background and a High-level Overview

Background

AngularJS1

  • Open-source JavaScript framework, maintained by Google
  • JavaScript MVC Framework for building Single-Page Applications (SPAs)
  • Notable emphasis on testability and embeddability
  • A framework: you either use it in full or not at all
  • MIT licensed

Similar Libraries and Frameworks

  • Backbone.js
    Minimalistic, low-level, make-your-own-stack library
  • Ember.js
    Minimal boilerplate, opinionated, "developer-friendly", very Angularish
  • Knockout
    MVVM framework. Declarative bindings, automatic UI refresh

2013 was the Trending Year of Angular

Google trending skyrocketing since early 2013
(230% increase in 01-07/2013)

...Yet new kids have been seen in the block already

  • React
    Virtual DOM based View implementation for JavaScript SPAs
  • Web Components
    Collection of Draft W3C specifications (Shadow DOM, Custom Elements, HTML Imports) to standardise SPAs. Multiple implementating libraries also available (Bosonic, Polymer, Polymer.Dart, X-Tag)

Design Overview

The Zen of Angular2

"Angular is built around the belief that declarative code is better than imperative when it comes to building UIs and wiring software components together, while imperative code is excellent for expressing business logic."

Design goals2

  • Decouple DOM manipulation from app logic (→ testability)
  • Regard application testing as equal in importance to application writing (testing difficult affects code structuring)
  • Decouple server side and client side (→ reusability, testability)
  • Guide developers (UI design → business logic → testing)

Use Cases

Angular Sweet Spot

"Angular was built for the CRUD application in mind. Luckily CRUD applications represent the majority of web applications." 2

Application examples:

  • The canonical AngularJS CRUD apps
  • http://builtwith.angularjs.org/project/Project-Time-Recorder


Use of AngularJS for GUI editors and games is specifically discouraged 2

HTML5/Hybrid Mobile Applications

  • SPAs are loaded only once → more native-like experience in mobile devices
  • Mobile Angular UI - a mobile-optimized version of Angular on top of Bootstrap 3 (improved performance, mobile-friendly controls with swipe support, offcanvas menus, etc.)
  • Ionic framework - AngularJS-based framework for creating hybrid mobile applications

Real-time Web Applications

Extend Angular's two-way data binding to server-side. Still a little bit experimental, but doable:

  • Hook up to socket.io interface with Angular Socket.io Seed
  • Use a Firebase backend with AngularFire


Application examples: chatroom, real-time TODO app


...But maybe just for real-time CRUD applications

When You Probably Should Not Use Angular

  • You already happily use another SPA framework :)
  • You need specific JavaScript components that are laborous to integrate with Angular
  • You need a lightweight application and/or need to make an embeddable application [6]
  • Generally when you are creating something that doesn't need Angular's core features (controllers, two-way data binding, etc.)

$ Prefix Indicates Angular Namespace

"To prevent accidental name collision, Angular prefixes names of objects which could potentially collide with $. Please do not use the $ prefix in your code as it may accidentally collide with Angular code."

  • Don't break the convention. Don't prefix your own objects with $.
  • Note that $ object usually refers to jQuery, not Angular

  • And the other way around: scope updates are reflected in DOM

    Using Controllers Correctly

    Do not use controllers for:

    • Direct DOM manipulation (→ Angular's data binding instead)
    • Input formatting (→ angular form controls)
    • Output formatting (→ filters)
    • Sharing stateless or stateful code (→ services)
    • Managing life-cycle of other components (→ injector)

    While any number of components (controllers, services, directives, etc.) can be bound to a single module, it is recommended to break your app into multiple modules:

    • A filter module, for filter declaration

    Expressions

    Expression

    • JavaScript-like code snippets, mainly used for data binding
    • Usually placed in bindings like: {{ expression }}
    • Are re-evaluated when bindings change

    Different from JavaScript Expressions

    • Evaluated against scope object (not window)
    • Silently handle undefined and null values
    • You cannot use control flow statements
    • Parsed with $parse, not with eval (=safer)

    Expression Examples

    Some simple math with values from scope:

    {{ savings + income - moneySpent }}
    {{ 3.14 * radius * radius }}

    String concatenation with null / undefined values:

    {{ "hello" + null + undefined + "world!" }}
    → "hellonull world!"

    Additional Notes

    • If you really need window object, you can use $window
    • If you really need eval, you can use $eval
    • Don't overdo expressions:
      • Filters for data manipulation
      • Directives for reusable components
      • Controllers for logic

    Filters

    • Filters perform data transformation (similar in spirit to UNIX pipe (|) filters
    • Filters can and are often chained
    • Examples:
      • filter a list by search string
      • format data according to a data type or locale
      • convert data into a specific format (e.g. json filter)

    Using Filters in View Templates

    Typical usage: apply filters to expressions in view templates:

    {{ expression | filter }}

    Filter chaining:

    {{ expression | filter1 | filter 2 }}

    With arguments

    {{ expression | filter1:arg1:arg2 }}

    Directives

    Directive

    • A behaviour or DOM transformation, which is triggered by the presence of a custom attribute, element name, class name or comment
    • Allows you to extend the HTML vocabulary in a declarative fashion.
    • Multiple directives have already been introduced
      (can you recall them?)
    • Simple example: ngHide = hide/show element based on expression
    • A more advanced example: make content editable with contentEditable="true" attribute.

    Directive Matching

    Example: 

    <input ng-click="action()">

    ng-click is matched with ngClick directive due to normalization:

    • Strip x- and data- from element/attribute name prefix
    • Convert ., - and _ delimited names to camelCase


    Best practice: use dash-delimited format in your names

    Directive Matching Types

    • E - Element names (<my-dir></my-dir>)
    • A - Attributes (<h1 my-dir="exp"><h1>)
    • C - Classes (<div class="my-dir"></div>)


    Note: prefer tag name directives

    Registering directive bindings (link)

    function link(scope, element, attrs) { ... }
    • scope is an Angular scope object.
    • element is the jqLite-wrapped element that this directive matches.
    • attrs is a hash object with key-value pairs of normalized attribute names and their corresponding attribute values.

    Server-side Communications

    $http

    • Rudimentary service for HTTP calls
    • Simply a wrapper around the browser's XMLHttpRequest object
    • Supported http-methods: GET, HEAD, POST, PUT, DELETE
    • Make requests using XHR objects or JSONP
    • Very similar to jQuery's ajax method
    • Based on deferred/promise APIs from $q

    Example

                      // inject $http
$http({method: 'GET', url: 'http://myservice/library/books'}).
  success(function(data, status, headers, config) {
      // this callback will be called asynchronously
      // when the response is available
  }).
  error(function(data, status, headers, config) {
      // called asynchronously if an error occurs
      // or server returns response with an error status.
  });

    Configuration object

    • Object given to $http when calling it as a method
    • Describe how to craft the XMLHttpRequest
    
var myPromise;
myPromise = $http({
    method: 'POST',                                 // http-method type
    url: 'http://myservice/library/books',          // URL for a resource
    params: {author: 'Jaakko Salonen'},             // queryparams
    data: {name: 'How to AngularJS', text: '...'},  // payload
    headers: {myheader: 'myheader'},                // headers for the call
    cache: true                                     // cache result for this request
});
//handle promise
                    
    http://myservice.com/library/books?author=Jaakko+Salonen
    • Also some other more advanced configuration options e.g. transformRequest and transformResponse

    Response object

    • Response to a $http call is a promise
    • Resolve and reject callback functions both get 4 properties (or parameters)
      • data - response payload
      • status - status number for the response
      • headers [function] - function to get the header value for given argument
      • config - configuration objects that was used to generate the original request

    You can also...

    • Use $http-services convenience methods to make your code a bit easier to read
      •

    • Interceptors - provide global functionality on all of our requests

    Case #2

    • Another part (in controller, factory, service - anything) of your application needs to know when resource is fetched
    • You don't know when it happens!
    • Using promises can solve this
    • Ok. With AngularJS there are technically other ways to solve this, but promises are the most elegant solution

    Meet $q

    • A helpfull service handling simultaneous rest calls and their responses
    • A promise/deferred implementation inspired by Kris Kowal's Q.
    • Part of the core of AngularJS

    Deferred objects

    Expose the associated Promise instance

    APIs for signaling the successful or unsuccessful completion

    Deferred API

    // Inject $q
var deferred = $q.defer();
deferred.promise; // Promise object bind to this deferred object
deferred.resolve(); // Resolves the promise and success callback gets executed
deferred.reject(); // Resolves the promise and reject callback gets executed
deferred.notify(); // Send notification

    Promises

    Retrieve by calling deferred.promise

    Allow interested parties to get access to the result of the deferred task when it completes

    Promises carry a few guarantees

    1. Only one resolve or reject will ever be called
    2. If the promise has been resolved or rejected, any handlers depending upon it will be called
    3. Handlers will always be called asynchronously - they won't block the rest of the app

    Promise API

    // Inject $q
var deferred = $q.defer();
var promise = deferred.promise;
//Use deferred api to resolve the promise

// regardless of when the promise was or will be resolved or rejected,
// then-method calls success or error callbacks asynchronously
// as soon as the result is available.
promise.then(successCallback, errorCallback, notifyCallback);

// shorthand for promise.then(null, errorCallback)
promise.catch(errorCallback);

// allows you to observe either the fulfillment or rejection of a promise,
// but to do so without modifying the final value.
promise.finally(callback);

    Basic example

    //assume $q and okToGreet are available (injected)
function asyncGreet(name) {
    var deferred = $q.defer();
    setTimeout(function() {
        deferred.notify('About to greet ' + name + '.');
        if (okToGreet(name)) {
            deferred.resolve('Hello, ' + name + '!');
        } else {
            deferred.reject('Greeting ' + name + ' is not allowed.');
        }
    }, 1000);
    return deferred.promise;
}

var promise = asyncGreet('Robin Hood');
promise.then(function(greeting) {
    alert('Success: ' + greeting);
}, function(reason) {
    alert('Failed: ' + reason);
}, function(update) {
    alert('Got notification: ' + update);
});

    Chaining

    Promises can be chained like callbacks from $resource

    But the code is cleaner and easier to understand

    Make more than one deferred objects and take their promises

    promiseB = promiseA.then(function(result) {
    return result + 1;
});

// promiseB will be resolved immediately after promiseA is resolved and its value
// will be the result of promiseA incremented by 1

    More

    Run a function when any number of promises in array have been resolved

    $q.all(promises).then(successFn, rejectFn))

    Make a promise that is resolved as rejected - used to forward rejection in a chain of promises

    $q.reject(message)

    $q.when(value) Wraps an object that might be a value or a (3rd party) then-able promise into a $q promise.

        promises.push(deferred.promise);
        MakeARestCall.put({data: item.oid}, function (success) {
            deferred.resolve(success);
        }, function (error) {
            deferred.reject(error);
        });
    });
    return promises;
};

var promises = doStuff();
$q.all(promises).then(function () {
    // all promises resolved
}, function () {
    // at least one promise was rejected
});

    Internet Explorer Compatibility

    • Add id="ng-app" to the element containing ng-app
    • If you use custom element tags (<ng:view>), extra boilerplate need to be added (IE < 8)
    • Polyfill JSON.stringify (IE7 and below)
    • Use ng-style tags for styles containing expressions (IE < 11)


    For details see http://docs.angularjs.org/guide/ie

    Ensuring JavaScript Code Quality

    ECMAScript 5 Strict Mode

    • A new feature in ECMAScript 5 that allows you to place a program, or a function, in a “strict” operating context.
    • Catches common coding "bloopers", throwing exceptions
    • Prevents or throws errors on unsafe actions
    • Disables confusing or poorly designed JavaScript features

    "use strict"

    Enable for whole script by adding in the beginning of the file:

    "use strict"

    Can be also used for individual functions:

    function myFunc() {
  "use strict";
}

    Catch: requires IE 10 or later (but does not break on older versions)

    Linting

    • Linting = checking your source code for style guideline compliance and suspicious usage of language
    • Especially important in dynamically typed languages like JavaScript where you lack the whole compile-time checking safety net
    • In addition works as a style guideline compliance checker

    jslint, jshint

    • Commonly used JavaScript linting tools
    • Choose one, add it to your build/CI-pipeline
    • You may end up disagreeing with the defaults
    • However: study to linting errors in depth before tweaking

    Minifying Angular Code

    Be Careful with Dependency Injection

    Be careful when using dependency injection in this fashion:

    angular.module('myApp').controller('MyControl', function ($scope, $http) {
  ...
});

    Minifiers can mangle object names, which break dependency injection (e.g. $scope → s)

    To stay minification-safe, write:

    angular.module('myApp').controller('MyControl', ['$scope', '$http', function ($scope, $http) {
  ...
})

    Minifiers don't mangle string values, ensuring injections work even with renamed parameter variables

    AngularJS Pre-minifier

    • You may still prefer or need to use the shorter form
    • Work around the problem by using a preminifier (ngmin)
    • Integrate to your build e.g. with grunt-ngmin or gulp-ngmin
